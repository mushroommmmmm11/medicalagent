## 🚀 **LLM 回答速度优化空间汇总**

### **1. 后端流式输出 (★★★★★ 优先级最高)**

**现状**: 前端支持流式但后端是同步阻塞

- ❌ AgentService.java：使用 `RestTemplate.exchange()` 同步等待
- ✅ 前端 ApiService.js 已支持流式接收

**优化方案**:

```
Java: RestTemplate → WebClient(反应式) + SSE 流式响应
预期收益: 3-5秒的LLM响应时间内，前端可提前看到前50%的内容
```

---

### **2. 知识库 RAG 缓存** (★★★★☆)

**现状**: OCR 有 Redis 缓存，RAG 没有

- ❌ rag.py：每次查询都返回新的检索结果，无缓存
- ❌ 向量数据库初始化缓慢（需要 Embeddings 调用）

**优化方案**:

```python
1. 加入 Redis 缓存层：cache_key = hash(query + model)
2. TTL 策略：医学知识 24 小时有效期
3. 预加载常见关键词的嵌入向量

预期收益: RAG 检索从 1-2s 降低到 100-300ms (命中缓存时)
```

---

### **3. 并行化工具调用** (★★★★☆)

**现状**: LangChain Agent 串行执行工具

- ❌ agent.py：`process_query()` 中 RAG + 用户历史查询是顺序执行
- ❌ 每个 Tool 调用都等待前一个完成

**优化方案**:

```python
# 改造为 asyncio.gather() 并行：
rag_result, user_history, report_classification = await asyncio.gather(
    retrieve_medical_knowledge(query),
    query_user_medical_history(user_id),
    classify_medical_report(query),
    return_exceptions=True
)

预期收益: 工具执行时间从 3秒串行 → 1秒并行 (提速 3倍)
```

---

### **4. 提前预取 OCR + 并发分析** (★★★☆☆)

**现状**: 顺序执行 (上传 → OCR → Agent)

- ✅ FileController.java 已支持 `prefetchOcr()`
- ❌ ChatWindow.vue 等 OCR 完成才调用 Agent

**优化方案**:

```javascript
// 改造前端流程：
1. 用户上传文件 → 立即触发 OCR 预取
2. 用户输入查询 → 同时调用 RAG + 等待 OCR 结果
3. 三者完成后统一发送给 Agent

// 调用链：
upload() → prefetchOcr() + RAG检索 // 并行
   ↓
chat() → 等同上面两个完成再调用 LangChain Agent

预期收益: 总耗时从传统串行的 OCR(3s) + Agent(5s) = 8s
        → 新模式的 max(OCR(3s), RAG(1s) + Agent(4s)) = 4s (提速 50%)
```

---

### **5. 模型选择优化** (★★★☆☆)

**现状**: 单一模型配置

- ❌ config.py：`DASHSCOPE_MODEL = "qwen3.5-plus-2026-02-15"` 固定
- ❌ main.py 多模型未充分利用

**优化方案**:

```yaml
# 新增模型选择策略：
fast_mode: "qwen-1.5-7b"      # 1-2s (医学问答)
balanced:  "qwen3.5-plus"    # 3-5s (诊断建议)
accurate:  "gpt-4-1106"      # 5-8s (复杂分析)

# 根据查询类型自动选择：
- 简单名词解释 → fast_mode
- 诊断分析 → balanced (当前默认)
- 多器官交互 → accurate

预期收益: 70% 的快速查询从 5s 降到 2s
```

---

### **6. 增量式 Token 优化** (★★★☆☆)

**现状**: 传入完整的 Prompt

- ❌ agent.py：完整 System Prompt + RAG 结果 + 用户历史都拼接到一个请求
- 这导致 Token 数膨胀可能

**优化方案**:

```python
1. 压缩 System Prompt：删除冗余的 [重要][关键] 重复标记
2. RAG Top-K 改为 2（从 3 降低），减少上下文
3. 用户历史：只发送最近 2 条而非所有

预期收益: Token 数减少 20-30%，API 响应时间减少 10-15%
```

---

### **7. WebClient 连接池优化** (★★☆☆☆)

**现状**: Java 已配置但可进一步优化

- ✅ OcrServiceClient.java：WebClient 已配置连接池
- ❌ Python OCR 和 LangChain 之间没有连接复用

**优化方案**:

```java
// 增加 keepalive & 连接复用
maxConnections: 50 → 100 (如果内存允许)
maxPendingAcquires: 250 → 500
tcpKeepAlive: true

预期收益: 高并发时连接建立延迟减少 30-50ms
```

---

### **8. 提前加载医学知识库** (★★☆☆☆)

**现状**: 启动时加载，但每次请求都需要检索

- ❌ rag.py：`_get_or_create_vectorstore()` 在初始化时加载

**优化方案**:

```python
# 在容器启动后立即预热：
- 加载 FAISS 索引到内存
- 预计算常见 10 个医学类别的向量
- 在 Redis 中存储热点查询的 Embedding

预期收益: 冷启动时第一次 RAG 查询从 3s → 0.5s
```
